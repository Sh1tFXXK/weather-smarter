# 环境驱动智能决策系统—开发文档

本项目从“天气查询”升级为 **环境驱动智能决策系统**。  
核心目标：多源环境数据 + 传统 NLP + 规则决策 + RAG 证据 + **LLM（必选）**，输出可解释建议。

当前版本说明：**LLM 必选且已部署**；其余接口为服务端模拟数据，不依赖外部天气服务。

以下内容已整理为 TODO-LIST，完成项已勾选。

## [x] 一、系统定位
- 系统名称：Weather Smarter
- 目标：将环境因素转化为可执行建议
- 价值：不仅回答天气，还对出行、运动、穿衣等决策给出可解释建议

## [x] 二、系统分层架构
```
用户输入
  ↓
传统 NLP（意图识别、槽位填充）
  ↓
环境数据获取（天气/空气质量/紫外线/时间）
  ↓
多源融合（环境快照）
  ↓
规则决策（可解释、稳定）
  ↓
LLM 增强（必选）
  ↓
结构化建议输出 + 记忆
```

## [x] 四、关键模块实现说明

### [x] 1. 传统 NLP 理解层
目的：体现“非纯调用 API”的技术实现。

- **意图识别**  
  - 使用 `TF-IDF + Linear SVM`，字符 n‑gram（2-4）  
  - 样本维护在 `backend/nlp/intent_data.py`  
  - 模型逻辑在 `backend/nlp/intent.py`

- **槽位填充**  
  - 规则抽取城市、时间、活动  
  - 实现在 `backend/nlp/slots.py`

### [x] 2. 环境数据获取
- 当前：服务端模拟数据（见 `backend/data/mock_data.py`）  
- 预留：`UAPIS /misc/weather`  
  - 实现：`backend/data/uapis_weather.py`
  - 支持重试与退避  

### [x] 3. 环境融合快照
模块：`backend/agent/environment_fusion.py`  
输出字段：
```
time, timeBucket, weather, comfortIndex, riskFlags, sources
```
用途：展示“环境感知 + 多源融合”的研究价值。

### [x] 4. RAG 检索增强
模块：`backend/rag/retriever.py`
- 统一知识库：`backend/rag/knowledge_base.jsonl`
- 基于 TF‑IDF（字符 n‑gram）构建向量检索
- `evidence` 作为可解释证据回传，并被 LLM 参考

### [x] 5. 规则决策引擎
模块：`backend/agent/decision_rules.py`
- 风险评分 + 场景规则（伞/运动/穿衣）
- 输出 `advice/riskLevel/reasons/actions`

### [x] 6. LLM 插拔层（必选）
模块：`backend/agent/llm_router.py`
- 当前版本必须启用 LLM
- 未来可通过 `enrich_decision()` 生成最终结构化建议  
- 输出格式：`{advice, riskLevel, reasons[], actions[]}`

### [x] 7. 对话记忆
模块：`backend/agent/memory_store.py`
- SQLite 存储会话记录  
- `sessionId` 有值时自动写入  
- 记忆接口：`/api/v1/memory/{session_id}`

## [x] 五、主要接口说明
详见 `docs/接口文档.md`，关键接口如下：
- `POST /api/v1/query`：NLP → 环境融合 → 规则 → RAG → **LLM（必选）**
- `GET /api/v1/misc/weather`：模拟环境数据入口
- `GET/DELETE /api/v1/memory/{session_id}`：上下文记忆
- `POST/GET/DELETE /api/v1/health/profile`：健康档案（模拟）
- `GET /api/v1/health/alerts/{user_id}`：健康提醒（模拟）
- `GET /api/v1/stocks/impact`：股市影响分析（模拟）
- `GET /api/v1/rag/evidence`：RAG 证据（模拟）
- `/`：简易前端测试页

## [x] 六、运行方式
1. 安装依赖
```
pip install -r requirements.txt
```
如需本地模型（llama.cpp）：
```
pip install -r requirements-local-llm.txt
```
2. 启动服务
```
py -m uvicorn main:app --reload --host 127.0.0.1 --port 8000
```
3. 访问
- 文档：`http://127.0.0.1:8000/docs`
- 测试页：`http://127.0.0.1:8000/`

本地模型部署详见：`docs/本地模型部署.md`

### [x] 启用 LLM（必选）
当前版本为模拟数据模式，但必须配置 LLM 才能调用 `/api/v1/query`。

Windows（cmd）示例：
```
setx LLM_PROVIDER openai
setx OPENAI_API_KEY "your_api_key_here"
setx LLM_MODEL gpt-4o-mini
```
重启服务后生效。

### [x] LM Studio（OpenAI 兼容）
如果你在 LM Studio 本地部署模型：
```
setx LLM_PROVIDER openai
setx OPENAI_BASE_URL http://127.0.0.1:1234/v1
setx OPENAI_COMPAT_MODE chat
setx OPENAI_API_KEY lm-studio
setx LLM_MODEL "你的模型名称"
```

### [x] 启用开源本地模型（推荐：llama.cpp 本地 GGUF）
适合“模型小、可本地运行、可微调”的需求，直接加载本地 GGUF 权重。

1. 安装依赖（可选独立文件）
```
pip install -r requirements-local-llm.txt
```

2. 下载 GGUF 模型（放在本地磁盘）
示例路径：
```
E:\models\qwen2.5-1.5b-instruct-q4_k_m.gguf
```

3. 配置环境变量（cmd）
```
setx LLM_PROVIDER llama_cpp
setx LLM_MODEL_PATH "E:\models\qwen2.5-1.5b-instruct-q4_k_m.gguf"
setx LLM_GPU_LAYERS 20
setx LLM_CTX 2048
```

### [x] 备选：Ollama 本地服务
适合希望独立运行模型服务的场景。
示例（cmd）：
```
setx LLM_PROVIDER ollama
setx OLLAMA_BASE_URL http://localhost:11434
setx LLM_MODEL llama3.2:1b
```

## [x] 七、配置项（环境变量）
说明：LLM 为必选，其余模拟数据不依赖外部服务。
- `UAPIS_BASE_URL`：上游天气服务地址  
- `UAPIS_TIMEOUT`：请求超时  
- `UAPIS_RETRIES`：重试次数  
- `UAPIS_BACKOFF`：退避秒数  
- `MEMORY_DB_PATH`：对话记忆数据库路径  
- `LLM_PROVIDER`：**必填**，`openai` 或 `ollama`  
- `OPENAI_API_KEY`：OpenAI API Key（SDK 自动读取）  
- `LLM_MODEL`：模型名（默认 `gpt-4o-mini`）  
- `LLM_TEMPERATURE`：温度（默认 `0.2`）  
- `LLM_MAX_TOKENS`：最大输出 token（默认 `300`）  
- `LLM_RESPONSE_FORMAT`：`json_object` 或 `json_schema`
- `RAG_KB_PATH`：知识库文件路径（默认 `backend/rag/knowledge_base.jsonl`）
- `RAG_TOP_K`：检索返回条数（默认 `3`）

## [x] 十、系统升级建议（面向复杂场景）
1. 重新包装定位为“环境驱动决策平台”，强调 **传统NLP→环境感知→规则引擎→RAG→LLM智能推理→记忆反馈** 的分层协同，并在文档/接口/前端描述中呈现这样的架构。
2. 扩展环境输入：接入 AQI、UV、交通拥堵、城市事件、用户日程/行程、物流状态等信号，统一映射到 `environment` 快照，为后续规则与 LLM 提供多源上下文。
3. 构建领域规则集（旅游/交通/物流/健康等），每条规则记录条件、风险等级与建议，将命中规则作为 evidence 返回，并作为 LLM 的显式上下文。
4. 强化提示词：让 LLM 输出固定结构（决策摘要+主要原因+风险提示+动作建议），并在 prompt 中提供规则/检索结果；在规则覆盖率低时由 LLM 补全、在结果里同时显示“规则结论 + LLM 补充”。
5. 在文档与前端测试页中增加场景示例（智慧城市调度、企业运营预警、健康提醒等），突出“多源 + 多层决策”的实际用途，便于科研/汇报展示。
6. 实用性补齐：将 `/forecast`、`/aqi`、`/tasks` 占位接口替换为真实数据源或明确标注为 stub，并在响应里返回数据源与更新时间。
7. 位置与时空能力升级：补充地理编码、IP 定位兜底、行政区纠错、时区处理、相对时间解析（今天/明早/下周末），缺失信息触发追问而不是直接猜测。
8. 预警与指数体系：引入极端天气/灾害预警、风寒/湿热/紫外线、空气污染等级等，并将其转化为可解释风险标签。
9. 场景化决策树：按通勤/运动/洗车/出行等场景调整规则权重与阈值，输出“是否建议 + 原因 + 替代方案”。
10. 时序决策输出：支持时间窗对比（早高峰/夜间/周末），展示风险随时间变化。
11. 多地点/路线差异：支持出发地/目的地/途经地的对比，按“最差点位”给出建议。
12. 多轮对话补齐：为缺失城市/时间/活动强度等信息设计澄清问题，提升结果确定性。
13. RAG 升级：从 TF-IDF 过渡到向量检索，引入权威知识与安全建议，并在证据中标注来源与可信度。
14. LLM 路由与成本控制：低风险场景走规则，高风险/复杂场景走 LLM，保证稳定性与成本。
15. 记忆体系升级：从“会话消息”扩展到“用户偏好/习惯”，设置 TTL 与容量上限，支持清理策略。
16. 任务与提醒闭环：天气触发提醒、降温/降雨/污染升级订阅，形成“建议→提醒→行动”体验。
17. 工程化与可运营：结构化日志、指标、链路追踪、错误码规范、缓存与降级策略、鉴权与限流。

## [x] 十一、健康能力设计（规划）
目标：把环境数据与个人健康风险建立可解释、可追溯的联系，提升实用性与场景深度。

### [x] 1. 健康档案（Health Profile）
- 与 `sessionId` 解耦，使用 `user_id` 或 `profile_id` 绑定。
- 仅在用户显式同意后保存；支持一键删除。
- 最小字段（MVP）：
  - `user_id`：用户标识
  - `conditions`：既往健康状况（枚举，如风湿/关节炎/哮喘/过敏性鼻炎/光敏）
  - `sensitivity`：可选阈值覆盖（对湿冷/高温/污染/紫外线敏感）
  - `consent`：同意存储标记与时间
  - `created_at` / `updated_at`

### [x] 2. 健康规则（Health Rules）
- 将环境快照映射为“健康风险触发器”，与 `conditions` 做匹配。
- 优先采用确定性阈值，便于解释与测试。
- 例（风湿/关节炎）：
  - 触发：`humidity >= 80` 且 `temp <= 12` 或 `precipitation > 0`
  - 建议：保暖、避免久坐潮湿环境、关节热敷、减少剧烈户外活动
- 例（哮喘/呼吸敏感）：
  - 触发：`AQI >= 100` 或 `wind_level >= 6`
  - 建议：降低户外强度、佩戴口罩、室内通风注意
- 例（光敏/皮肤问题）：
  - 触发：`UV >= 7`
  - 建议：加强防晒、避免正午暴晒

### [x] 3. 输出结构（建议新增）
- `healthAlerts`: `[{condition, riskLevel, reasons, actions}]`
- `healthRiskLevel`: `low|medium|high`
- `followUps`: 当健康档案缺失或信息不足时，提示补全

### [x] 4. 安全与合规
- 明确“非医疗建议，仅供参考”，禁止诊断或用药建议。
- 记录用户授权与删除路径，避免长期存储隐私数据。

### [x] 5. 代码落地建议
- 新模块：`backend/agent/health_rules.py`
- 新存储：`backend/agent/health_profile_store.py`
- API 扩展：`/api/v1/health/profile`
- 测试：`tests/test_health_rules.py`、`tests/test_health_profile_store.py`

### [x] 6. 与 LLM 的协同
- 规则先行，LLM 仅做语言与解释增强。
- LLM 输出限制为 JSON 结构，避免扩展到“医疗建议”。

## [x] 十二、优化路线（里程碑）

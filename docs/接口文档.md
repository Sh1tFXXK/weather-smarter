# Weather Smarter 接口文档（v1）

定位：环境驱动智能决策系统（Environment-Driven Decision System）。  
核心能力：融合环境数据 + 传统 NLP + 规则决策 + RAG 证据 + **LLM（必选）**，输出可解释建议。  
当前版本：**LLM 必选且已部署**；其余接口为服务端模拟数据，不依赖外部天气服务。

## 基础信息
- Base URL（本地）：`http://127.0.0.1:8000`
- 版本：所有路由以 `/api/v1` 开头
- Content-Type：请求 `application/json`
- 文档：`/docs`、`/redoc`、`/openapi.json`
- 前端测试页：`/`（简易 UI）

## 全局约定
- 时间：ISO-8601，示例 `2026-03-14T08:00:00+08:00`
- 语言：`lang=zh|en`（天气接口），`locale` 用于 UI 语言偏好
- 会话：`sessionId` 可选，存在则记忆上下文
- Mock：除 LLM 外，当前所有接口返回模拟数据；字段结构与未来真实接口保持一致
- 错误：
  - 上游错误：`{"detail": {"code": "...", "message": "..."}}`
  - 校验错误（FastAPI 默认）：`{"detail": [...]}`

## 1) 自然语言决策
POST `/api/v1/query`

用途：  
自然语言 → 意图识别/槽位填充 → 环境融合 → 规则决策 → RAG 证据 → **LLM 决策（必选）** → 结构化建议。

请求体：
```json
{
  "query": "明天北京适合跑步吗",
  "locale": "zh-CN",
  "lang": "zh",
  "sessionId": "demo-session",
  "city": "北京",
  "adcode": null,
  "location": {"name": "北京", "lat": 39.9, "lon": 116.4},
  "time": "2026-03-14T08:00:00+08:00",
  "scenario": null,
  "units": "metric"
}
```

响应字段说明：
- `intent`：意图分类结果
- `nlu`：传统 NLP 结果（意图 + 槽位）
- `environment`：环境融合快照（多源环境感知）
- `evidence`：RAG 检索证据（知识库片段）
- `decision`：LLM 决策结果（基于规则与证据）
- `memory`：若 `sessionId` 存在，返回最近对话记录
- `healthAlerts`：健康风险提醒（基于健康档案或关键词）
- `healthRiskLevel`：健康风险等级 `low|medium|high`

## 2) 环境天气（模拟）
GET `/api/v1/misc/weather`

用途：  
按 `adcode` 或 `city` 查询实时天气与扩展指标。当前为模拟数据。

查询参数：
- `adcode`：行政区编码（优先级高）
- `city`：城市名（支持中英文）
- `lang`：`zh|en`
- `extended`：返回扩展字段（体感、能见度、气压、紫外线、AQI 等）
- `forecast`：返回多天预报（最多 7 天）
- `hourly`：逐小时预报（24 小时）
- `minutely`：分钟级降水（仅国内城市）
- `indices`：18 项生活指数

示例：
```bash
curl "http://127.0.0.1:8000/api/v1/misc/weather?city=北京&extended=true&indices=true"
```

说明：模拟数据不会返回上游错误。

## 3) 会话记忆
GET `/api/v1/memory/{session_id}`

查询参数：
- `limit`：返回条数，范围 1-200，默认 20

DELETE `/api/v1/memory/{session_id}`  
用途：清空某个会话的上下文记忆。

## 4) 健康档案（模拟）
POST `/api/v1/health/profile`  
GET `/api/v1/health/profile/{user_id}`  
DELETE `/api/v1/health/profile/{user_id}`

说明：存储在本地 SQLite（模拟数据），服务重启可保留。

字段说明（请求/响应）：
- `user_id`：用户标识
- `conditions`：健康状况列表（如 `rheumatism`/`asthma`/`photosensitivity`）
- `note`：备注
- `consent`：同意存储
- `sensitivity`：敏感阈值覆盖（如 `{ "humidity": 70, "uv": 6 }`）
- `created_at` / `updated_at`：时间戳

## 环境融合字段说明
`environment` 是系统的“环境感知快照”，用于学术展示与可解释决策。
- `time`：当前/事件时间
- `timeBucket`：时间段（morning/noon/afternoon/evening/night）
- `weather`：聚合后的气象数据（温度、体感、湿度、风、降水、UV、AQI）
- `comfortIndex`：0-100 简易舒适指数
- `riskFlags`：风险标签（rain/snow/storm/air_quality/uv/temperature_extreme/wind）
- `sources`：融合来源列表

## RAG 说明
- 知识库位置：`backend/rag/knowledge_base.jsonl`
- 检索方法：TF-IDF 字符 n‑gram + 余弦相似度
- `evidence` 用于提供可解释依据，LLM 会参考该字段生成建议

## 环境变量
说明：LLM 为必选，其余模拟数据不依赖外部服务。
- `UAPIS_BASE_URL`：上游天气服务地址（默认 `https://uapis.cn/api/v1`）
- `UAPIS_TIMEOUT`：请求超时秒数（默认 `5.0`）
- `UAPIS_RETRIES`：重试次数（默认 `2`）
- `UAPIS_BACKOFF`：退避秒数（默认 `0.6`）
- `MEMORY_DB_PATH`：记忆数据库路径（默认 `backend/data/memory.db`）
- `LLM_PROVIDER`：**必填**，`openai` / `ollama` / `llama_cpp`
- `LLM_MODEL_PATH`：本地 GGUF 模型路径（`LLM_PROVIDER=llama_cpp`）
- `LLM_LORA_PATH`：LoRA 适配器路径（可选）
- `LLM_CTX`：上下文长度（llama.cpp）
- `LLM_GPU_LAYERS`：GPU 层数（llama.cpp）
- `LLM_THREADS`：CPU 线程（llama.cpp）
- `LLM_BATCH`：批大小（llama.cpp）
- `OPENAI_BASE_URL`：OpenAI 兼容服务地址（如 LM Studio）
- `OPENAI_COMPAT_MODE`：`chat` 表示使用 Chat Completions 兼容模式
- `OPENAI_API_KEY`：OpenAI API Key
- `LLM_MODEL`：模型名（默认 `gpt-4o-mini`）
- `LLM_TEMPERATURE`：温度（默认 `0.2`）
- `LLM_MAX_TOKENS`：最大输出 token（默认 `300`）
- `LLM_RESPONSE_FORMAT`：`json_object` 或 `json_schema`
- `OLLAMA_BASE_URL`：本地开源模型服务地址（默认 `http://localhost:11434`）
- `OLLAMA_TIMEOUT`：本地模型超时（秒）
- `RAG_KB_PATH`：知识库文件路径（默认 `backend/rag/knowledge_base.jsonl`）
- `RAG_TOP_K`：检索返回条数（默认 `3`）

## 系统升级建议
1. 把系统位置表述为“环境驱动决策平台”，文档/前端中突出 NLP→环境融合→规则→RAG→LLM→记忆 的分层流水线。
2. 扩展环境信号（AQI/UV/交通/城市事件/日程）给 `environment`，确保规则与 RAG、LLM 都在同一上下文下运行。
3. 构建按领域（旅行/物流/健康）分类的规则库，记录命中规则编号与描述并作为 evidence 反馈，便于审计与 LLM 推理。
4. 提示词要求 LLM 产出固定结构（摘要+原因+风险+动作），把规则/检索结果一并放入 prompt，必要时输出“规则结论 + LLM 补充”。
5. 文档和测试页新增智慧城市、企业运营、健康管理等场景示例，强调多源多层决策对实际业务的价值。
6. 接口实用性补齐：将 `/forecast`、`/aqi`、`/tasks` 的示例接口标注为 stub 或替换为真实实现，并在响应中增加 `source` 与 `updatedAt` 字段。
7. 时空能力强化：新增“时间窗对比”“多地点对比”参数，支持出发地/目的地/途经地的风险差异化输出。
8. 交互补齐：当城市/时间/活动缺失或置信度低时返回 `followUps` 引导补齐信息，避免错误建议。
9. 版本化与兼容：对重要字段（如 `decision`、`environment`）做版本标识，保障客户端演进安全。
10. 稳定性与安全：统一错误码、增加 `traceId`、鉴权与限流策略；对上游不可用时返回可解释降级结果。
